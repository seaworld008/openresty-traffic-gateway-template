# OpenResty 自建网关设计说明

## 目标

构建一套面向中小团队、偏生产实践的 OpenResty 单节点网关模板。要求是仓库可以直接克隆、快速配置、通过 Docker Compose 启动，并且后续维护成本低、扩展路径清晰。

## 范围

- 单台 Linux 主机部署
- 使用官方 `openresty/openresty` 镜像，并固定完整 tag
- ACME 证书管理独立放在 `ssl/` 目录
- 按域名拆分站点配置文件
- 代理、TLS、安全头、缓存策略使用可复用 snippets
- 自带本地验证用示例后端
- 提供部署、配置、维护、排障文档

## 推荐方案

仓库采用清晰的分层结构：

- 一个 `docker-compose.yml` 管理网关服务和本地验证示例后端
- 一个 `openresty/` 目录负责运行时配置、缓存、日志和站点文件
- 一个 `ssl/` 目录负责 certbot 状态、ACME challenge 目录和证书脚本
- 一个 `examples/` 目录放本地联调样例和 curl 测试说明

这样做的好处是职责清楚：

- 网关配置统一在 `openresty/`
- 证书生命周期统一在 `ssl/`
- 验证示例统一在 `examples/`

## 架构说明

### 生产流量路径

`Internet -> OpenResty (80/443) -> 内部 upstream 服务`

OpenResty 作为唯一公网入口，负责：

- HTTP 跳 HTTPS
- TLS 终止
- 域名路由
- 反向代理
- 基础安全头
- 请求大小与超时控制
- 访问日志与错误日志
- 可选的静态资源缓存

### 本地验证路径

`curl / 浏览器 -> OpenResty -> 示例后端容器`

本地验证默认自带示例后端，因此仓库即使没有真实业务应用，也能完成完整链路测试。

## 关键设计决策

### 1. 站点配置按域名拆分

每个域名一个文件，放在 `openresty/conf.d/sites/` 下。这样可以缩小改动范围，减少合并冲突，也便于单域名回滚。

### 2. 通用逻辑收敛到 snippets

代理头、TLS 参数、安全头、缓存策略统一放在 `openresty/snippets/`。这样可以避免复制粘贴漂移，后续做统一加固更安全。

### 3. SSL 生命周期独立管理

ACME 状态、证书和脚本都放到 `ssl/` 下，运维排障时不需要在网关配置目录里到处寻找。

### 4. 优先固定镜像 tag

固定 tag 能让升级、回滚和排障都具备可重复性。README 会明确说明为什么不能使用浮动 tag。

### 5. 支持“零业务依赖”的本地烟雾测试

仓库直接附带示例后端，这样执行 `docker compose up -d` 后就可以立刻用 `curl` 验证完整网关行为。

## 运维模型

### 启动流程

1. 复制 `.env.example` 为 `.env`
2. 按需要调整邮箱、域名等参数
3. 启动示例后端和 OpenResty
4. 本地模式先生成自签证书
5. 生产模式再申请 ACME 证书并 reload

### 续签流程

证书续签通过独立脚本执行。续签成功后，由单独的 reload 脚本触发 OpenResty 重载。后续可以很自然地挂到宿主机 cron 或 systemd timer 上。

### 排障重点

重点围绕以下几类问题给出文档与命令：

- 配置语法校验
- 容器健康状态和日志
- 证书签发失败
- upstream 连通性问题
- HTTP / HTTPS 验证命令

## 风险与缓解措施

- ACME 签发依赖真实公网 DNS 和 80 端口可达
  - 缓解方式：提供本地自签证书流程，并在文档中明确生产前提
- 单节点天然存在单点故障
  - 缓解方式：文档中说明未来如何扩展到双机或负载均衡前置
- 跨容器 reload 处理容易做得很隐蔽
  - 缓解方式：保持 reload 脚本简单、显式、可人工执行

## 成功标准

- 仓库可以直接克隆并本地启动
- 容器内 `openresty -t` 能通过
- 示例站点能正确路由到示例后端
- 本地 HTTPS 能通过自签证书跑通
- 生产证书签发流程有清晰文档和脚本
- 文档足以支持部署、日常维护和新项目复用
